## Context

本變更實作拜拜儀式的完整流程，是應用的核心功能。使用者可選擇「拜神明」或「祭拜祖先/親人」兩種模式，系統根據模式提供對應的步驟引導。

本變更假設 project-setup 已完成，包含 Vue 3 + Vite 7 環境、shadcn-vue + Tailwind CSS 4 設計系統、Pinia 狀態管理、檔案式路由骨架、靜態資料結構等基礎設施皆已就緒。

**目標使用者**：台灣地區的一般民眾，以手機使用為主。

**技術限制**：
- 前端為主的 PWA 架構
- 需支援行動裝置相機（QR Code 掃描、供品拍照）
- 焚香背景計時需在使用者繼續操作時持續運作

## Goals / Non-Goals

**Goals:**
- 實作兩種祭拜模式的完整儀式流程（拜神明 / 祭祖）
- 提供步驟進度指示器，讓使用者清楚知道目前進度
- 焚香為非阻塞步驟，點香後可立即進入祈禱或留下觀看動畫
- 使用 Pinia 管理儀式狀態，已登入使用者刷新頁面可恢復進度
- 儀式完成後顯示完整摘要，已登入使用者自動保存紀錄

**Non-Goals:**
- 不實作焚香動畫本身（由 incense-simulation 變更負責）
- 不實作金紙焚燒動畫（由 joss-paper-burning 變更負責）
- 不實作供品擺設互動（由 offering-arrangement 變更負責）
- 不實作擲筊互動（由 divination-blocks 變更負責）
- 不實作紙紮物品焚燒（由 paper-craft-burning 變更負責）
- 不實作使用者驗證邏輯（由 user-auth 變更負責）
- 不實作環保成效計算邏輯（由 eco-impact-tracker 變更負責）

## Decisions

### 1. 儀式狀態管理：Pinia Store + IndexedDB 持久化

**選擇**：使用單一 `useWorshipStore` 管理整個儀式流程狀態，搭配 IndexedDB 為已登入使用者持久化進度。

**理由**：
- 儀式流程是線性的多步驟操作，單一 Store 可集中管理當前步驟、各步驟資料、焚香計時等狀態
- IndexedDB 可儲存較大的結構化資料（如供品照片 blob），比 localStorage 更適合
- 焚香結束時間戳存入 IndexedDB，刷新後可計算剩餘時間恢復計時

**替代方案考量**：
- 多個 Store 分別管理各步驟：增加複雜度，步驟間的依賴與流轉不好處理
- localStorage：容量限制（5MB），不適合存放供品照片等二進位資料

### 2. 兩種模式共用流程引擎

**選擇**：以統一的流程引擎驅動兩種模式，根據模式類型動態決定步驟序列。

**理由**：
- 拜神明流程：掃描廟宇 QR Code → 選擇神明 → 擺設供品 → 焚香 → 祈禱 → 擲筊 → 燒金紙
- 祭祖流程：輸入對象稱謂 → 選填祭拜地點 → 記錄供品（拍照） → 焚香 → 祈禱 → 燒金紙 → 燒紙紮物品
- 兩種流程有共用步驟（焚香、祈禱、燒金紙），統一引擎可減少重複邏輯
- 步驟序列以陣列定義，方便未來新增或調整步驟順序

### 3. 焚香非阻塞設計

**選擇**：焚香步驟完成點香後立即解鎖下一步，計時在背景持續運作。

**理由**：
- 真實拜拜中，點香後就可以開始祈禱，不需等香燒完
- 使用者體驗更流暢，不會卡在等待畫面
- 背景計時使用 `setTimeout` + 結束時間戳（而非 `setInterval`），刷新後可根據時間戳恢復

### 4. 步驟導航策略：Vue Router + Store 同步

**選擇**：每個步驟對應一個路由（`/worship/incense`、`/worship/prayer` 等），步驟切換透過 `router.push` 導航，同時更新 Store 中的當前步驟。

**理由**：
- 每個步驟有獨立 URL，支援瀏覽器上一頁/下一頁
- 路由守衛可防止使用者跳過步驟（例如直接訪問 `/worship/prayer` 而未完成前置步驟）
- 與 project-setup 定義的路由骨架一致，直接實作已有的佔位頁面

**替代方案考量**：
- 單頁面 + 步驟元件切換：URL 不變，無法使用瀏覽器導航，不利於刷新恢復

### 5. 已登入 vs 訪客的差異處理

**選擇**：儀式流程本身對所有使用者一致，差異僅在持久化策略。

**理由**：
- 已登入使用者：儀式進度持久化至 IndexedDB，刷新可恢復；完成後自動保存至帳號
- 訪客使用者：儀式進度僅存於記憶體（Pinia Store），刷新後重新開始
- 這樣的設計讓儀式流程邏輯不需要區分身份，持久化邏輯獨立處理

## Risks / Trade-offs

### [Risk] 子功能模組尚未實作時的整合
本變更依賴多個子功能模組（焚香動畫、金紙焚燒、供品擺設、擲筊、紙紮焚燒），這些模組可能尚未實作。
-> **Mitigation**：各步驟頁面預留插槽（slot）或元件介面，子功能未就緒時顯示簡化的佔位 UI，待對應變更實作後替換。

### [Risk] IndexedDB 相容性與容量
部分舊版瀏覽器或隱私模式下 IndexedDB 可能不可用。
-> **Mitigation**：使用 `try-catch` 偵測 IndexedDB 可用性，不可用時 fallback 至 localStorage（僅存文字資料）或純記憶體模式。

### [Risk] 焚香計時精確度
瀏覽器背景 tab 可能會限制計時器精度。
-> **Mitigation**：使用結束時間戳（`Date.now() + duration`）而非累計計時，每次進入頁面時重新計算剩餘時間，不依賴計時器的連續精度。

### [Risk] 文化敏感度
儀式流程涉及宗教信仰，UI/UX 設計與文案需莊重得體。
-> **Mitigation**：遵循 project-setup 建立的廟宇莊嚴風格設計系統，文案用語謹慎，避免戲謔。
