# 設計文件：紙紮物品焚燒功能

## 設計背景

本功能為「祭拜祖先/親人」模式的專屬環節，讓使用者可以選擇或自訂紙紮物品進行虛擬焚燒。設計上需要提供直覺的物品分類瀏覽與選擇機制，並以 Canvas 粒子動畫呈現焚燒效果。此功能與既有的燒金紙功能共用動畫基礎架構，但紙紮物品的焚燒視覺表現需有所區別。

前置依賴：`setup-project-rules` 已完成專案基礎建設。

## 設計目標

- 提供清晰的紙紮物品分類與品項瀏覽介面，方便使用者快速找到想焚燒的物品
- 支援多選與自訂物品，彈性滿足不同使用者需求
- 複用既有 Canvas 粒子引擎，擴充紙紮物品專屬的焚燒動畫參數
- 紙紮物品焚燒動畫與金紙焚燒有明確視覺差異（物品外觀、燃燒方式、灰燼效果）
- 動畫等級設定與既有系統共用，確保低階裝置也能順暢體驗

## 非目標

- 不實作紙紮物品的 3D 模型或 WebGL 渲染——以 2D 圖示搭配 Canvas 粒子效果呈現
- 不實作紙紮物品的購買或解鎖機制——所有預設物品均免費使用
- 不處理紙紮物品的圖片上傳——自訂物品統一使用通用紙紮外觀
- 不在非祭祖模式下顯示此功能

## 設計決策

### 決策 1：紙紮物品資料以靜態 JSON 管理

**選擇**：將預設紙紮物品分類與品項定義為靜態 JSON 檔案

**理由**：
- 紙紮物品清單為固定內容，不需要動態載入
- 靜態 JSON 可隨版本控制，方便日後新增品項
- 避免為靜態資料建置額外的後端 API

**替代方案**：
- 後端 API 動態管理：增加複雜度，但品項不常變動，不值得
- 硬編碼於元件中：不利維護與擴充

### 決策 2：焚燒清單使用 Pinia Store 管理

**選擇**：以 Pinia store 集中管理焚燒清單狀態（已選物品、自訂物品）

**理由**：
- 物品選擇頁面與焚燒動畫頁面需共享清單狀態
- Pinia 響應式狀態方便 UI 即時反映選擇變更
- 與既有 `useAuthStore` 等 store 架構一致

### 決策 3：複用金紙焚燒的 Canvas 粒子引擎

**選擇**：擴充既有粒子引擎，新增紙紮物品專屬的粒子參數組

**理由**：
- 避免重複建構粒子系統，減少維護成本
- 紙紮物品焚燒的核心動畫機制（粒子生成、運動、消亡）與金紙相同
- 透過不同參數組（粒子大小、顏色、燃燒速度、灰燼形狀）區分視覺效果

**替代方案**：
- 獨立建構紙紮專用動畫系統：重複工作，且未來維護成本翻倍

### 決策 4：自訂物品使用通用紙紮外觀

**選擇**：自訂物品不要求使用者上傳圖片，統一以通用紙紮外觀加上文字標籤呈現

**理由**：
- 降低使用者操作門檻，輸入名稱即可
- 避免圖片上傳帶來的儲存與審核問題
- 通用紙紮外觀搭配文字標籤已足夠傳達意涵

## 風險與緩解

| 風險 | 影響 | 緩解措施 |
|------|------|----------|
| 紙紮物品分類過多導致瀏覽體驗不佳 | 使用者找不到想要的物品 | 限制初期分類數量（5-7 類），提供搜尋或常用物品推薦 |
| 紙紮焚燒動畫與金紙焚燒視覺差異不明顯 | 使用者感知不到功能區分 | 紙紮物品使用較大的物件外觀、不同的燃燒色調（偏暖橘紅）與獨特的灰燼飄散效果 |
| 自訂物品名稱可能包含不當內容 | 潛在內容風險 | 初期以前端基本過濾處理，後續可加入雲端內容審核 |
| 粒子引擎擴充後效能下降 | 焚燒動畫卡頓 | 紙紮物品焚燒同樣受動畫等級設定控制；避免同時渲染過多粒子 |
