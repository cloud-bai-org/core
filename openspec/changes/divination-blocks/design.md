## Context

擲筊是台灣民間信仰中與神明溝通的核心方式，信眾將兩個半月形筊杯拋出，依據落地朝向判讀神明旨意。數位擲筊需要在視覺與互動上還原儀式感，同時確保結果的隨機公平性。

前置依賴：`setup-project-rules` 已完成專案基礎建設。

## Goals

- 以物理動畫還原筊杯拋擲的真實感與儀式感
- 結果判定清晰明確，圖示與文字並呈
- 隨機結果符合傳統機率分佈（聖筊 50%、笑筊 25%、陰筊 25%）
- 支援同一場儀式中多次擲筊與歷程紀錄
- 動畫播放期間鎖定操作，避免重複觸發

## Non-Goals

- 3D 物理引擎模擬（過度複雜，行動裝置效能負擔大）
- 擲筊音效（後續迭代再加入）
- 擲筊結果與籤詩連動（屬於抽籤功能範疇）
- 擲筊紀錄跨場儀式彙整（歷程僅限當前儀式）

## Decisions

### Decision 1：動畫實作方式
**選擇**：CSS/JS 動畫搭配預設關鍵影格模擬物理效果

**理由**：
- 相較 Canvas 或 WebGL，CSS/JS 動畫在行動裝置上效能更佳且實作成本低
- 筊杯動畫為固定路徑（拋起、翻轉、落地），不需要即時物理運算
- 可透過 GSAP 或 CSS keyframes 實現流暢的翻轉與落地效果

**替代方案**：
- Canvas 2D 物理模擬：開發成本高，筊杯造型繪製複雜
- 3D WebGL（Three.js）：效能負擔大，行動裝置相容性風險高
- 預錄影片：檔案大、無法動態對應結果

### Decision 2：隨機結果產生機制
**選擇**：先決定結果，再播放對應動畫

**理由**：
- 結果先決定可確保機率精確控制（聖筊 50%、笑筊 25%、陰筊 25%）
- 動畫只需對應三種落地姿態，簡化動畫系統設計
- 避免即時物理模擬的不可預測性導致機率偏差

**替代方案**：
- 即時物理模擬後判讀結果：機率難以精確控制，需大量校調

### Decision 3：擲筊狀態管理
**選擇**：以 Pinia composable 管理擲筊狀態（isAnimating、currentResult、history）

**理由**：
- 與既有 Nuxt 3 / Pinia 架構一致
- 響應式狀態方便 UI 即時反映動畫播放中/結果/歷程
- 歷程紀錄為陣列結構，方便在儀式結束時彙整

### Decision 4：觸發方式
**選擇**：同時支援按鈕點擊與觸控向上滑動手勢

**理由**：
- 按鈕點擊為基礎互動，確保所有裝置可用
- 向上滑動手勢模擬實體擲筊動作，增強儀式沉浸感
- 兩種方式共用同一觸發函式，實作成本低

## Risks / Trade-offs

- **動畫流暢度風險**：低階行動裝置可能因 CSS 動畫過於複雜而卡頓。緩解措施：動畫保持簡潔，控制同時動畫元素數量，必要時提供簡化版動畫。
- **手勢衝突風險**：向上滑動手勢可能與頁面捲動衝突。緩解措施：手勢偵測限定在擲筊區域內，並設定最小滑動距離閾值。
- **隨機性感知問題**：使用者可能覺得結果「不夠隨機」（如連續出現同一結果）。緩解措施：使用加密安全的隨機數產生器（crypto.getRandomValues），結果純粹由機率決定，不做人為干預。
