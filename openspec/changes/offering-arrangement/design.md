# 設計文件：offering-arrangement

## 背景脈絡

供品擺設是台灣民間信仰拜拜儀式的核心環節。本變更需同時支援兩種使用情境：

- **拜神明模式**：提供虛擬供品庫與供桌拖放擺設，適用於無法親臨現場的線上拜拜
- **兩種模式皆適用**：實體供品拍照紀錄、QR Code 掃描關聯、計時連動推播通知

系統需在行動裝置上提供流暢的拖放互動體驗，同時整合相機、QR Code 掃描與推播通知等原生能力。

## 目標

- 提供直覺的虛擬供品瀏覽與選擇介面
- 實作流暢的拖放擺設體驗，支援觸控裝置
- 整合相機 API 進行實體供品拍照紀錄
- 支援 QR Code 掃描以關聯實體供桌與儀式
- 焚香計時結束時自動推播通知使用者領取供品

## 非目標

- 不處理供品的電商購買流程
- 不實作 3D 供桌渲染（使用 2D 俯視圖即可）
- 不處理多人同時擺設同一供桌的即時同步
- 不實作供品的 AI 自動推薦

## 設計決策

### 決策 1：拖放實作方式

**選擇：使用原生 HTML Drag and Drop API 搭配觸控事件 polyfill**

- 桌面端使用原生 HTML5 Drag and Drop API
- 行動端使用 touch event（touchstart / touchmove / touchend）實作拖放
- 供桌採用 2D 俯視角度的 canvas 或絕對定位佈局
- 供品位置以相對座標（百分比）儲存，確保不同螢幕尺寸的一致性

**替代方案（未採用）**：
- 使用第三方拖放函式庫（如 dnd-kit）：增加套件相依性，但若專案已使用 React 生態系可考慮
- 使用 canvas 自行繪製：實作成本高，互動事件處理複雜

### 決策 2：相機 API 整合

**選擇：使用 Web API（MediaDevices / input capture）搭配原生橋接備援**

- 優先使用 `<input type="file" accept="image/*" capture="environment">` 觸發系統相機
- 若需即時預覽，使用 `navigator.mediaDevices.getUserMedia()` 取得相機串流
- 拍攝的照片壓縮後上傳至物件儲存（如 S3 / GCS）
- 儲存照片 URL 與儀式 ID 的關聯

**替代方案（未採用）**：
- 純原生 APP 相機模組：需分別實作 iOS / Android，維護成本高

### 決策 3：QR Code 掃描

**選擇：使用 Web-based QR Code 掃描器（如 jsQR 或 html5-qrcode）**

- 透過 `getUserMedia` 取得相機串流，搭配 QR Code 解碼函式庫即時掃描
- QR Code 內容格式：包含供桌識別碼與位置資訊的結構化 URL
- 掃描成功後自動將供桌位置關聯至當次儀式紀錄

**QR Code 內容格式設計**：
```
https://{domain}/altar/{altar_id}?location={location_id}
```

### 決策 4：推播通知與計時連動

**選擇：後端排程觸發推播**

- 焚香計時由後端管理倒數時間
- 計時結束時由後端觸發推播通知（FCM / APNs）
- 通知內容：「儀式已完成，請前往供桌領取供品」
- 前端同時顯示 in-app 通知作為備援

## 資料模型概要

- **Offering**：供品主檔（名稱、分類、圖片、描述）
- **OfferingCategory**：供品分類（水果、三牲、糕點、飲品、鮮花等）
- **CeremonyOffering**：儀式與供品的關聯（儀式 ID、供品 ID、位置座標、類型：虛擬/實體）
- **OfferingPhoto**：實體供品照片（儀式 ID、照片 URL、拍攝時間）

## 風險

| 風險 | 影響 | 緩解措施 |
|------|------|----------|
| 行動裝置拖放體驗不一致 | 不同瀏覽器/裝置的觸控事件行為差異大 | 充分測試主流裝置，必要時提供非拖放的替代擺設方式（點選位置） |
| 相機權限被拒絕 | 使用者無法拍照記錄供品 | 提供明確的權限說明與手動上傳照片的備援方案 |
| QR Code 掃描在低光源環境失敗 | 廟宇環境可能光線不足 | 提供手動輸入供桌編號的備援方式 |
| 推播通知未送達 | 使用者未收到儀式完成通知 | 同時提供 in-app 通知、SMS 備援 |
| 供品圖片資源量大 | 初期需準備大量高品質供品素材 | 分階段上架，優先處理最常見的供品分類 |
