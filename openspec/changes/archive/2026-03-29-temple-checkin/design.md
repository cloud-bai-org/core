# 設計文件：廟宇打卡功能

## 設計背景

本功能將雲端拜拜應用從純線上體驗擴展至線下實體廟宇互動。使用者可透過 QR Code 掃描或手動搜尋在實體廟宇完成打卡，並結合祈願目的獲得參拜路線推薦。已登入使用者可累積參拜足跡與成就徽章，形成長期參與的激勵機制。

前置依賴：`project-setup` 已完成專案基礎建設。

## 設計目標

- 使用 html5-qrcode 函式庫實作 QR Code 掃描，辨識廟宇 ID 完成打卡
- 提供手動搜尋作為掃描失敗的替代方案，從靜態 JSON 資料檔搜尋廟宇
- 打卡後引導使用者選擇祈願目的，依據廟宇神明產生推薦參拜路線
- 以地圖元件呈現已登入使用者的參拜足跡
- 建立成就徽章系統，統計參拜歷程並在達成條件時頒發徽章
- 所有廟宇資料使用自訂虛構名稱，避免涉及真實廟宇

## 非目標

- 不實作即時定位驗證（不要求使用者必須在廟宇附近才能打卡）
- 不實作社群分享功能（打卡分享至社群平台留待後續變更）
- 不處理 QR Code 的產生與列印——僅處理掃描端
- 不實作廟宇評論或評分功能
- 不使用真實廟宇名稱

## 設計決策

### 決策 1：QR Code 掃描使用 html5-qrcode

**選擇**：使用 html5-qrcode 函式庫進行 QR Code 掃描

**理由**：
- 純 JavaScript 實作，無需原生 App 支援，適合 Web App
- 支援行動裝置相機存取，相容性良好
- 社群活躍、文件完善，整合成本低

**替代方案**：
- ZXing.js：功能較全面但體積較大，對本場景過於複雜
- 原生相機 API 自建解析：開發成本高且相容性處理繁瑣

### 決策 2：廟宇資料使用靜態 JSON

**選擇**：將廟宇資料（名稱、位置、神明、殿堂）以靜態 JSON 檔案儲存於前端

**理由**：
- 初期廟宇數量有限，無需建置後端 API 與資料庫
- 靜態 JSON 可與應用一同部署，離線也能搜尋
- 未來擴展時可平滑遷移至後端 API

**替代方案**：
- 後端 API + 資料庫：初期開發與維運成本過高
- CMS 管理：增加系統複雜度，目前不需要

### 決策 3：參拜路線基於靜態規則產生

**選擇**：根據祈願目的與廟宇神明的對應關係，以靜態規則產生推薦參拜順序

**理由**：
- 參拜順序有傳統依據（如先拜天公爐、再依殿堂順序），規則明確
- 靜態規則簡單可靠，不需要 AI 或複雜演算法
- 規則資料可與廟宇 JSON 一同維護

**替代方案**：
- AI 動態推薦：增加後端依賴與延遲，初期不必要
- 使用者自訂路線：增加介面複雜度，可作為未來功能

### 決策 4：足跡地圖使用輕量地圖元件

**選擇**：使用 Leaflet + OpenStreetMap 呈現足跡地圖

**理由**：
- 開源免費，無 API key 費用限制
- 輕量且行動裝置效能良好
- 支援標記（marker）與自訂圖標，足以呈現打卡位置

**替代方案**：
- Google Maps：需 API key 且有使用量費用
- Mapbox：功能強大但同樣有費用考量

### 決策 5：成就徽章以前端靜態規則判定

**選擇**：徽章達成條件在前端以靜態規則判定

**理由**：
- 條件簡單明確（如：打卡 5 間廟宇、完成所有祈願類型等）
- 前端即時判定可立即顯示動畫回饋，不需等待後端回應
- 徽章定義與條件資料以 JSON 管理，易於擴充

## 風險與緩解

| 風險 | 影響 | 緩解措施 |
|------|------|----------|
| 行動裝置相機權限被拒絕 | 無法使用 QR Code 掃描 | 提供手動搜尋作為替代方案；掃描前顯示相機權限說明 |
| html5-qrcode 在部分瀏覽器相容性問題 | 掃描功能失效 | 偵測瀏覽器支援度，不支援時直接引導至手動搜尋 |
| 靜態 JSON 廟宇數量增長後搜尋效能下降 | 搜尋回應變慢 | 初期數量有限無影響；未來可改用後端 API 或 Web Worker 處理 |
| 地圖元件在低效能裝置上載入緩慢 | 足跡頁面體驗不佳 | 延遲載入地圖元件；標記數量過多時做叢集處理 |
| 成就徽章條件前端判定可被竄改 | 徽章真實性存疑 | 目前為個人記錄性質，不涉及獎勵兌換；未來可加入後端驗證 |
