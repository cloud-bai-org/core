## Context

雲端拜拜應用的核心理念是零門檻體驗，使用者不需要註冊即可完成拜拜儀式。同時為了跨裝置同步與長期資料保存，需要提供第三方登入機制。台灣市場 Google 與 LINE 為最主流的登入方式，覆蓋率最高。

前置依賴：`setup-project-rules` 已完成專案基礎建設。

## Goals

- 訪客模式零門檻進入，不阻斷核心拜拜流程
- 支援 Google OAuth 與 LINE Login 兩種第三方登入
- 訪客升級為登入用戶時，本機資料可無縫合併至雲端
- 訪客與正式帳號紀錄在雲端明確區隔
- 登出後回到訪客模式，不遺失本機紀錄

## Non-Goals

- 自建帳號密碼系統
- Apple Sign-In（非原生 App，體驗較差）
- Facebook 登入（台灣使用率持續下降）
- 多帳號合併（同一使用者用 Google 和 LINE 登入視為不同帳號，未來再處理）

## Decisions

### Decision 1：三層驗證策略
**選擇**：訪客模式（零門檻）+ Google OAuth + LINE Login

**理由**：
- 訪客模式確保核心功能零摩擦，降低跳出率
- Google 與 LINE 在台灣市場覆蓋率最高，兩者互補
- 委託第三方 OAuth 省去自建驗證系統的維運成本

**替代方案**：
- 自建帳密系統：開發與維護成本高，密碼安全責任大，使用者註冊意願低
- Apple Sign-In：Web App 體驗不佳，台灣 Android 用戶比例高
- Facebook 登入：台灣年輕族群使用率持續下滑

### Decision 2：訪客紀錄儲存策略
**選擇**：本機 localStorage/IndexedDB 為主，同步上傳雲端並標記為「訪客」

**理由**：
- 本機儲存確保離線可用與即時回應
- 上傳雲端方便開發者分析行為與除錯
- 訪客身份標記讓雲端資料可區分正式帳號

**替代方案**：
- 僅本機儲存：無法做行為分析，清除瀏覽器資料即遺失
- 僅雲端：離線無法使用，增加延遲

### Decision 3：前端狀態管理使用 Pinia useAuthStore
**選擇**：以 Pinia store 集中管理驗證狀態（登入狀態、使用者資訊、token）

**理由**：
- Nuxt 3 生態系標準方案，與 SSR 相容
- 響應式狀態方便 UI 即時反映登入/登出
- 可搭配 `pinia-plugin-persistedstate` 持久化 token

### Decision 4：訪客升級合併策略
**選擇**：登入時將本機訪客紀錄合併至雲端帳號

**理由**：
- 使用者在訪客模式累積的紀錄不應因登入而消失
- 合併後本機訪客紀錄標記為已同步，避免重複

## Risks / Trade-offs

- **訪客紀錄遺失風險**：使用者清除瀏覽器資料會遺失未登入的本機紀錄。緩解措施：儀式完成後提示登入綁定。
- **第三方 SDK 依賴**：Google OAuth 與 LINE Login SDK 版本更新可能造成相容性問題。緩解措施：封裝 auth service 層，隔離 SDK 細節。
- **訪客合併衝突**：若同一裝置有多位訪客使用，合併時可能混入他人紀錄。緩解措施：合併前顯示紀錄預覽讓使用者確認。
- **LINE Login 限制**：LINE Login 無法取得 Email，使用者資料較 Google 少。影響：帳號識別需以 LINE userId 為主鍵。
